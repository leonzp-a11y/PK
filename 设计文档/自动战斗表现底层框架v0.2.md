# 自动战斗表现底层框架 v0.2

## 1. 文档目标

本文档用于定义当前项目中“自动战斗”模式下的战斗表现底层框架。它不直接讨论技能表、配置表或具体技能字段，而是优先拆解战斗表现层的组成部分，明确每一部分应该解决什么问题，以及程序、前端、美术在表现实现中需要共同遵循的底层规则。

当前战斗底层逻辑为：

- 敌人按照自身攻击节奏自动行动。
- 玩家技能按照各自 CD 自动释放。
- 玩家主要承担观察、理解、成长选择和战局判断，而不是实时手动释放每一个技能。

因此，表现层的首要职责不是“响应玩家输入”，而是让玩家持续看懂自动战斗的因果：

```text
谁在行动
-> 行动影响了谁
-> 为什么发生了伤害/控制/治疗/格挡
-> 当前状态如何改变
-> 战斗节奏正在走向哪里
```

## 2. 自动战斗表现核心原则

自动战斗中，玩家不能依靠“我刚按了什么”来理解战斗，所以表现必须主动解释战场。

所有战斗表现优先回答以下问题：

1. 当前是谁在行动？
2. 这次行动的目标是谁？
3. 这次行动是普通事件、威胁事件，还是阶段事件？
4. 行动是否命中，结果是什么？
5. 目标产生了什么后果或状态变化？
6. 战斗节奏是否进入了新阶段？

底层原则：

- 表现层必须解释自动演算，而不是只播放特效。
- 所有重要行动都要有可识别主体。
- 所有伤害、控制、治疗、格挡、死亡都要能被玩家理解来源和结果。
- 大面积特效必须让位于“行动主体、命中对象、结果反馈、状态变化”。
- 普通循环、爆发、危机、Boss 阶段、奖励阶段必须有节奏差异。

## 3. 信息优先级

自动战斗的信息优先级如下：

| 优先级 | 信息 | 说明 |
|---|---|---|
| P0 | 即将发生的威胁 | 敌人攻击、控制、Boss 大招、危险状态 |
| P1 | 当前行动主体 | 此刻是谁在出手，玩家技能还是敌人技能 |
| P2 | 行动目标 | 这次行动影响谁 |
| P3 | 行动结果 | 伤害、治疗、格挡、闪避、控制、死亡 |
| P4 | 状态变化 | 冻结、中毒、蛛网、增益、减益、护盾 |
| P5 | 节奏变化 | 一波敌人结束、卡牌选择、Boss 阶段、奖励 |
| P6 | 氛围强化 | 粒子、屏幕覆盖、装饰性动作、环境光效 |

底层规则：

- P6 不能遮挡 P0-P5。
- 当前行动主体必须先于技能特效被识别。
- 行动结果不能只靠飘字表达。
- 状态变化不能只靠战场特效表达，必须与 UI 状态互相印证。

## 4. 战斗表现层拆分

自动战斗表现层建议拆成 7 个部分：

```text
自动战斗表现层
├─ 1. 行动表现
├─ 2. 威胁表现
├─ 3. 命中表现
├─ 4. 受击/后果表现
├─ 5. 状态表现
├─ 6. UI/飘字表现
└─ 7. 节奏/阶段表现
```

这 7 个部分是底层表现框架，不绑定具体武器、法术、怪物或 Boss。后续所有冰系、火系、毒系、物理攻击、蜘蛛怪、Boss 技能，都应该在这 7 个部分上扩展。

## 5. 行动表现

### 5.1 目标

让玩家知道“现在是谁在行动”。

自动战斗中，玩家不一定手动触发技能，因此每一次重要行动都要通过表现告诉玩家：

- 是玩家侧技能在释放。
- 是敌人正在攻击。
- 是 Boss 正在准备大招。
- 是某个状态或被动正在触发。

### 5.2 表现内容

行动表现包括：

- 玩家技能释放前的蓄力或起手。
- 敌人攻击前的动作前摇。
- Boss 大招前的阶段提示。
- 自动技能 CD 到点后的释放节奏。
- 多个行动同时发生时的主次排序。

### 5.3 底层规则

- 每次重要行动都必须有行动主体识别。
- 玩家技能和敌人攻击不能在同一视觉层级上同时抢主表现。
- 多个玩家技能连续触发时，需要有主次节拍，而不是所有技能同时爆亮。
- 敌人攻击前摇必须比普通待机动作更清楚。
- 如果行动只是低价值循环事件，可以弱化表现，但不能完全失去来源。

### 5.4 程序关注点

- 表现层需要知道当前行动主体。
- 自动触发技能如果同帧集中发生，需要支持表现错峰或表现队列。
- 敌人技能需要为前摇表现留出时间窗口。
- 行动事件应能区分普通行动、强行动、Boss 行动、阶段行动。

### 5.5 前端关注点

- 可以用行动条、CD 完成闪烁、目标高亮、技能图标激活辅助理解。
- UI 提示不应抢过战场主体，但要解释技能为什么释放。
- 自动技能释放时，技能图标应有“即将释放/正在释放/进入冷却”的状态差异。

### 5.6 美术关注点

- 玩家技能起手、敌人前摇、Boss 前摇需要有不同动作语言。
- 自动战斗里，“谁在动”比“特效炫不炫”更重要。
- 敌人前摇应有清晰剪影变化，而不是只靠小特效。

## 6. 威胁表现

### 6.1 目标

让玩家知道“危险要来了”。

自动战斗中，威胁表现不是为了让玩家实时躲避，而是为了解释战斗因果：

- 为什么玩家掉血。
- 为什么玩家被控。
- 为什么战斗节奏被打断。
- 为什么某个敌人需要优先关注。

### 6.2 表现内容

威胁表现包括：

- 敌人普通攻击前摇。
- 敌人强攻击提示。
- 控制技能预兆。
- Boss 大招预兆。
- 场地危险预兆。
- 玩家受击来源提示。

### 6.3 三段式规则

所有敌方伤害或控制事件必须遵循：

```text
预兆 -> 生效 -> 结果
```

预兆：告诉玩家危险即将发生。  
生效：告诉玩家危险正在发生。  
结果：告诉玩家自己受到了什么影响。

### 6.4 底层规则

- 所有敌方伤害/控制都必须有“预兆、生效、结果”。
- 控制技能和 Boss 技能前摇必须强于普通攻击。
- 威胁预兆不能只靠小图标表达，必须有战场内表现。
- 自动战斗中，威胁表现的重点是“因果解释”，不是“操作反应”。
- 玩家受伤必须能看出来源方向或状态来源。

### 6.5 程序关注点

- 敌方技能需要有可表现的前摇窗口。
- 如果数值结算是立即的，也应补一个视觉预兆或延迟表现窗口。
- Boss 大招、控制、强攻击应能向表现层暴露威胁等级。

### 6.6 前端关注点

- 玩家受伤、被控、被减益时，需要有来源和状态提示。
- Boss 大招可以有屏幕上方或中心短提示，但不能长期遮挡战场。
- 控制状态应同步影响技能按钮、状态图标或玩家状态区。

### 6.7 美术关注点

- 敌人前摇要有清晰剪影变化。
- 蛛网、毒液、冲撞、爆炸等危险类型要有各自可识别语言。
- Boss 大招前兆应具备环境或镜头层面的压迫感。

## 7. 命中表现

### 7.1 目标

让玩家知道“这次行动打中了谁”。

命中表现是当前 demo 最需要强化的模块。技能特效可以很强，但玩家必须先看清：

- 命中对象是谁。
- 接触点在哪里。
- 是否命中、格挡、闪避、未命中或免疫。
- 群体技能实际影响了哪些目标。

### 7.2 表现内容

命中表现包括：

- 命中点。
- 命中目标高亮。
- 命中特效。
- 命中音效。
- 命中瞬间的短暂停顿或节奏强调。
- 未命中/格挡/免疫的差异表现。

### 7.3 底层规则

- 命中点优先于大面积特效。
- 命中对象必须在短时间内可识别。
- 群体技能要能区分“范围覆盖”和“实际命中目标”。
- 未命中、格挡、免疫不能使用正常命中表现。
- 飘字不能单独承担命中确认。

### 7.4 程序关注点

- 表现层需要知道实际命中目标，而不只是技能范围。
- 多目标命中最好允许错峰表现，避免所有数字和特效同帧堆叠。
- 命中结果需要区分正常命中、格挡、闪避、免疫、暴击、击杀。

### 7.5 前端关注点

- 伤害数字不能是唯一命中确认。
- 目标血条变化、状态图标、目标高亮要辅助命中识别。
- 多目标伤害数字需要排序、合并或节流，避免堆叠。

### 7.6 美术关注点

- 命中特效应短、准、聚焦。
- 持续特效应降级，不能盖住命中对象。
- 冰、火、毒、物理等不同类型应有不同接触语言。

## 8. 受击/后果表现

### 8.1 目标

让玩家知道“这次命中造成了什么后果”。

自动战斗中，玩家关注的不只是单次伤害数字，而是战局走势。因此受击后果必须表达：

- 目标是否被削弱。
- 目标是否被打断。
- 目标是否进入状态。
- 目标是否死亡或进入下一阶段。

### 8.2 表现内容

受击/后果表现包括：

- 受击动作。
- 硬直。
- 击退。
- 冻结。
- 破防。
- 死亡。
- 掉落。
- Boss 局部反馈。

### 8.3 底层规则

- 普通敌人必须有基础受击反应。
- 重要命中必须有明显后果。
- Boss 可以不击退，但必须被影响。
- 死亡表现必须清理画面，为下一段节奏让路。
- 受击后果强度要和事件价值匹配，不能所有命中都同等强。

### 8.4 程序关注点

- 命中结果需要驱动表现，不应只驱动数值。
- Boss 需要支持局部受击或状态层表现。
- 普通受击、重受击、控制受击、击杀应能触发不同表现模板。

### 8.5 前端关注点

- 血条变化要和受击后果同步。
- 死亡、奖励、阶段完成要有清晰确认。
- Boss 血条、阶段条和状态图标要与战场表现一致。

### 8.6 美术关注点

- 小怪受击、精英受击、Boss 受击应分级。
- 控制状态不能只靠颜色滤镜，要有结构变化。
- Boss 受击建议使用局部反应：头部、腹部、腿部、甲壳、眼睛。

## 9. 状态表现

### 9.1 目标

让玩家知道“当前谁处于什么状态，状态是否还在”。

自动战斗里状态可能很多，如果状态表现不清晰，玩家会不知道为什么某个单位突然不能动、持续掉血、回血、格挡或反弹。

### 9.2 表现内容

状态表现包括：

- 状态进入。
- 状态持续。
- 状态刷新。
- 状态叠层。
- 状态结束。

### 9.3 三段式规则

所有状态必须包含：

```text
进入 -> 持续 -> 结束
```

进入：状态刚刚发生。  
持续：状态仍然存在。  
结束：状态已经解除。

### 9.4 底层规则

- 所有状态必须有“进入、持续、结束”。
- 状态持续表现要低于命中表现。
- 状态图标和战场表现要互相对应。
- 同类状态要有统一视觉语言。
- 重要状态刷新或叠层时，要有轻量反馈。

### 9.5 程序关注点

- 表现层需要知道状态进入、刷新、结束，而不只是当前是否存在。
- 叠层和持续时间如果影响战局，需要暴露给 UI。
- 状态事件需要区分敌人状态和玩家状态。

### 9.6 前端关注点

- 状态图标需要分优先级，不要堆成一排看不懂。
- 玩家身上的负面状态比敌人身上的小状态优先级更高。
- 持续时间、层数、可解除状态需要被清楚表达。

### 9.7 美术关注点

- 冻结、中毒、蛛网、燃烧、护盾要有可区分语言。
- 状态持续特效应克制，不要和命中特效抢层级。
- 状态结束要有清楚收束，避免玩家误判状态仍在。

## 10. UI/飘字表现

### 10.1 目标

让玩家用最少阅读成本理解战斗结果。

当前 demo 中，伤害数字、格挡、反弹、未命中、回血、状态文字同时出现时，信息容易拥挤。UI 和飘字必须服务阅读，而不是增加噪声。

### 10.2 表现内容

UI/飘字包括：

- 伤害数字。
- 治疗数字。
- 格挡/闪避/未命中。
- 状态文字。
- 血条。
- CD。
- 卡牌选择。
- 奖励提示。

### 10.3 底层规则

- 飘字是结果确认，不是唯一反馈。
- 同屏飘字必须分层：玩家受伤 > 控制/死亡 > 暴击/重击 > 普通伤害。
- 同类小数字可合并或弱化。
- 玩家状态和敌人状态不能混用同一位置语言。
- 字号、颜色、轨迹和停留时间必须按事件价值分级。

### 10.4 程序关注点

- 多段小伤害可以合并显示或节流。
- 同帧多事件需要 UI 优先级排序。
- 结果事件应能区分伤害、治疗、格挡、闪避、免疫、状态进入、状态结束。

### 10.5 前端关注点

- 飘字轨迹、字号、颜色、停留时间要分级。
- 玩家受击数字应和敌人受击数字区分位置与颜色。
- 技能 CD、状态图标和战斗飘字之间要避免互相遮挡。

### 10.6 美术关注点

- 字效和 UI 动效要服务阅读，不要只追求弹跳感。
- 暴击、击杀、控制需要有独立字效。
- 普通小数字应克制，为高价值结果留表现空间。

## 11. 节奏/阶段表现

### 11.1 目标

让玩家理解战斗正在进入哪一段。

自动战斗非常依赖节奏理解。如果每一秒都是技能、特效和数字，玩家会疲劳，也无法判断当前是普通循环、爆发、危机、收尾还是奖励。

### 11.2 表现内容

节奏/阶段表现包括：

- 开战。
- 第一轮自动技能释放。
- 敌人反扑。
- 玩家压制。
- Boss 阶段。
- 卡牌选择。
- 奖励。
- 战斗结束。

### 11.3 底层规则

- 战斗需要峰谷，不应全程高强度。
- 选卡/奖励是节奏断点，需要清晰但不拖沓。
- Boss 阶段变化必须明显高于普通技能。
- 新卡/成长后的第一次释放要有验证反馈，让玩家知道成长生效了。
- 长时间持续特效要给高潮留空间。

### 11.4 程序关注点

- 自动释放技能可以做表现错峰。
- 阶段切换需要明确事件通知。
- 奖励、选卡、Boss 阶段应作为表现阶段事件，而不是普通 UI 弹窗。

### 11.5 前端关注点

- 卡牌选择、奖励、阶段提示要与战场节奏衔接。
- 不要让 UI 弹窗完全切断玩家对战局的理解，除非这是明确回合断点。
- 新卡获得后的第一次技能释放，应在 UI 或技能图标上有强化反馈。

### 11.6 美术关注点

- 普通循环、爆发、Boss 大招、胜利奖励应有不同视觉强度。
- 阶段转换可以调用环境、镜头、灯光、敌人动作强化感知。
- 奖励表现要清理战斗噪声，给玩家一个明确收束。

## 12. 当前 demo 的优先落点

基于蜘蛛洞穴录屏，第一轮优先落点建议如下：

### 12.1 命中清晰度

问题：

- 冰系大面积特效容易盖住敌人、命中点和血条。

改动方向：

- 缩短命中爆点生命周期。
- 降低持续冰雾透明度。
- 给实际命中目标增加短暂描边或闪白。
- 群体命中数字错峰或合并。

### 12.2 敌方威胁前摇

问题：

- 蛛网、毒液、Boss 攻击结果明显，但前摇和来源不够清楚。

改动方向：

- 蜘蛛控制技能增加明显预兆。
- Boss 大招增加环境或身体部位变化。
- 玩家受击增加来源方向和状态提示。

### 12.3 受击后果

问题：

- 小怪和 Boss 受击主要靠数字表达，身体反馈弱。

改动方向：

- 小怪增加基础受击动作。
- 重要命中增加硬直、冻结或局部反应。
- Boss 增加局部受击反馈，不要求击退。

### 12.4 UI/飘字降噪

问题：

- 普通伤害、格挡、反弹、回血、状态文字同时堆叠。

改动方向：

- 按事件价值设置飘字优先级。
- 多段小伤害合并或弱化。
- 玩家受伤与敌人受伤分开显示语言。

### 12.5 自动战斗节奏

问题：

- 中后段技能循环视觉强度持续偏高，缺少峰谷。

改动方向：

- 自动技能错峰释放。
- 普通循环降低表现强度。
- Boss 大招、击杀、奖励保留更高表现峰值。

## 13. v0.2 硬规则清单

第一版执行时，建议先锁定以下硬规则：

1. 所有重要行动必须能识别行动主体。
2. 所有敌方伤害/控制必须有“预兆、生效、结果”。
3. 所有玩家技能命中必须有“命中点、目标反馈、结果确认”。
4. 所有状态必须有“进入、持续、结束”。
5. 飘字只能作为结果确认，不能代替命中和受击表现。
6. 大面积持续特效不得遮挡敌人剪影、血条、玩家状态和技能 CD。
7. 普通自动循环事件不得使用 Boss/暴击/击杀级别的表现强度。
8. 同一时刻屏幕中央只能有一个主表现事件。
9. Boss 可以不被击退，但必须通过局部反应表现“被影响”。
10. 卡牌/成长后的第一次表现必须让玩家看懂“成长生效了”。
