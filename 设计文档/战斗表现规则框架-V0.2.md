# [PK] 战斗表现规则框架 V0.2

本框架用于统一自动战斗中的表现设计语言。它优先定义表现模块、信息层级和底层规则，不绑定具体技能表、配置表、武器、法术或怪物类型。

当前战斗系统建立在两个基本前提之上：

- 己方技能按冷却时间自动释放。
- 敌方按自身攻击节奏自动行动。
- 玩家主要通过观察战斗过程、理解结果反馈，并据此选择成长路径。

## 1. 框架目标与战斗前提

### 1.1 表现层核心目标

- 看清谁在行动：明确区分己方与敌方行为主体。
- 看清行动影响谁：清晰展示技能或攻击的作用目标。
- 看清命中与结果：即时反馈伤害、治疗、状态、格挡、闪避、击杀等效果。
- 看清状态变化：实时反映 Buff、Debuff、控制、DOT、护盾等持续状态。
- 看清战斗节奏变化：直观呈现波次、Boss 转阶段、卡牌选择、胜利/失败等关键节点。

### 1.2 框架阅读方式

本文档按“模块规则 + 横向规则”组织：

- 行动表现模块：解决己方和敌方“正在做什么”。
- 信息反馈模块：解决屏幕上的信息“放在哪里、谁优先显示”。
- 结果表现模块：解决命中、失败、防御、DOT、受击绑定等“结果怎么表现”。
- 全局表现调度：解决镜头、节奏、阶段断点等“战斗整体如何被观看”。

### 1.3 通用原则

- 基础信息由动作、动效、结果反馈和 UI 完成。
- 高价值事件才调用镜头、节奏、强化飘字等全局表现。
- 同一规则只在一个主章节定义，其他章节只引用或说明触发关系。
- 普通事件必须克制，为暴击、击杀、Boss 转阶段等关键事件留表现空间。

## 2. 行动表现模块

行动表现模块负责解释战斗中“谁在行动、行动是否生效、行动影响了谁”。该模块分为己方行动表现和敌方行动表现。

### 2.1 己方行动表现

己方行动表现负责向玩家解释：己方角色正在做什么、技能是否生效，以及这次行动的价值是什么。

#### 2.1.1 模块总览

| 项目 | 内容 |
| --- | --- |
| 模块目标 | 让玩家看懂己方技能正在自动释放，并理解这次技能的生效点和价值。 |
| 画面关注点 | 底部技能 UI、己方手部/武器/施法动作、敌方命中点、敌方结果飘字。 |
| 表现链路 | 冷却完成 → 起手/准备 → 施法/生效 → 命中反馈 → 冷却/收束。 |
| 前端关注 | 技能图标状态、冷却/吟唱/持续施法状态、释放触发时机、命中事件同步。 |
| 美术关注 | 起手动作、释放特效、命中特效、Buff/Debuff 进入反馈、持续施法状态。 |
| 配套图示 | 己方技能释放链路图；飘字位置/颜色/优先级图。 |
| 产出目标 | 前端能知道每个技能阶段要触发什么表现；美术能知道每类技能需要补哪些动作/特效资产。 |

#### 2.1.2 技能释放表现流程

```mermaid
flowchart TD
    A["技能 CD 完成"] --> B["进入释放准备"]
    B --> C["行动主体识别"]
    C --> D["技能起手动作"]
    D --> E{"技能类型"}

    E --> E1["瞬发技能"]
    E --> E2["吟唱/蓄力技能"]
    E --> E3["持续施法技能"]
    E --> E4["Buff/强化技能"]

    E1 --> F1["释放瞬间动效"]
    E2 --> F2["吟唱读条 + 蓄力动效"]
    E3 --> F3["持续施法动作 + 持续动效"]
    E4 --> F4["强化进入动效 + 状态 UI"]

    F1 --> G["作用目标识别"]
    F2 --> G
    F3 --> G
    F4 --> G

    G --> H["命中/生效反馈"]
    H --> I["结果反馈"]
    I --> J["技能进入冷却"]
    J --> K["行动收束"]
```

#### 2.1.3 分层规则

动作层：

- 包含普通攻击动作、技能释放动作、技能持续施法动作、技能被打断动作、己方受伤动作、行动恢复动作。
- 重要技能必须有可识别的起手动作。
- 持续施法必须能看出“仍在施法”的状态。
- 行动结束需要有明确收束，避免技能表现之间相互干扰。
- 受伤或被控制状态不能与释放动作表现产生冲突。

动效层：

- 包含瞬发技能动效、持续施法动效、蓄力/吟唱动效、Buff 增强动效、Debuff 施加动效、持续区域动效。
- 瞬发技能强调释放瞬间的视觉爆发。
- 持续施法强调过程的稳定状态。
- Buff/Debuff 需要有进入反馈，不能仅依赖 UI 图标。
- 持续动效不能长期遮挡主要目标和关键 UI 信息。

结果层：

- 包含命中特效、命中目标反馈、伤害/治疗飘字、状态进入提示、击杀/奖励反馈、轻微震动或节奏停顿。
- 命中点的表现优先级高于大面积范围特效。
- 飘字不能作为唯一反馈手段，必须结合其他表现。
- 群体技能要清晰区分范围覆盖特效和实际命中目标反馈。
- 击杀、暴击、控制等高价值事件的表现强度必须高于普通命中。

### 2.2 敌方行动表现

敌方行动表现负责向玩家解释：敌人要做什么、攻击何时生效，以及玩家为什么受伤或被控。

#### 2.2.1 模块总览

| 项目 | 内容 |
| --- | --- |
| 模块目标 | 让玩家看懂敌人正在攻击、攻击何时生效，以及己方为什么掉血或被控。 |
| 画面关注点 | 敌人身体动作、攻击特效、生效帧、己方受击反馈、己方结果飘字。 |
| 表现链路 | 敌方行动触发 → 前摇/准备 → 攻击生效 → 己方受击确认 → 行动收束。 |
| 前端关注 | 敌方行动事件、攻击生效帧、己方受击事件、状态变化和 UI 同步。 |
| 美术关注 | 普通攻击/强攻击前摇差异、命中特效、受击反馈、死亡/收束动作。 |
| 配套图示 | 敌方攻击表现三段式图；己方结果飘字示意图。 |
| 产出目标 | 前端能把敌方行为和己方受击事件对齐；美术能按攻击强度制作可读前摇和命中反馈。 |

#### 2.2.2 攻击表现流程

```mermaid
flowchart TD
    A["敌方行动触发"] --> B["敌方主体识别"]
    B --> C{"行动类型"}

    C --> C1["普通攻击"]
    C --> C2["强攻击"]
    C --> C3["控制技能"]
    C --> C4["特殊攻击"]

    C1 --> D1["普通前摇"]
    C2 --> D2["强化前摇"]
    C3 --> D3["控制预兆"]
    C4 --> D4["特殊预兆"]

    D1 --> E["攻击生效"]
    D2 --> E
    D3 --> E
    D4 --> E

    E --> F{"是否命中己方"}
    F --> F1["命中"]
    F --> F2["未命中/闪避/抵抗"]

    F1 --> G["己方受击反馈"]
    F1 --> H["血量/状态变化"]
    F2 --> I["失败结果反馈"]

    G --> J["UI/飘字确认"]
    H --> J
    I --> J

    J --> K["敌方行动收束"]
```

#### 2.2.3 分层规则

动作层：

- 包含待机动作、普通攻击前摇、强攻击前摇、技能释放动作、受伤动作、死亡动作、Boss 阶段动作。
- 所有敌方攻击必须有可识别的前摇动作。
- 强攻击前摇必须在时长、幅度或特征上区别于普通攻击。
- 控制技能或 Boss 技能的前摇规格必须更高。
- 敌人受伤、死亡动作要能有效清理战场上的冗余战斗信息。

威胁动效层：

- 包含攻击预警、控制预警、范围预警、弹道/飞行物、屏幕边缘危险、Boss 大招预兆。
- 威胁预兆不能只依赖 UI 小图标，必须在战场上有明确视觉表现。
- 控制类技能必须通过预兆让玩家提前感知危险类型。
- Boss 大招必须拥有独立且高规格的预兆表现。
- 威胁表达需要让玩家理解“危险从哪里来”。在视野受限或第一人称/近背视角下，威胁来源可转化到屏幕空间表达，例如屏幕边缘光效、方向性镜头晃动、特效从对应方向进入、受击音效方向感等。详细视角适配规则后续单独整理。

结果层：

- 包含己方受击反馈、己方掉血飘字、己方状态变化、红屏/屏幕边缘反馈、技能按钮受限、玩家血条变化。
- 己方受伤反馈必须能让玩家看出伤害来源。
- 被控、中毒、减速等状态需同时具备战场视觉反馈和 UI 图标反馈。
- 玩家受击飘字与敌方受击飘字需要遵循信息反馈模块中的屏幕分区和优先级规则。

## 3. 信息反馈模块

信息反馈模块负责解释自动战斗系统正在执行什么，以及结果如何影响战局。它不创造战斗事件，而是把战斗事件翻译成可读的屏幕信息。

### 3.1 UI 信息表现

#### 3.1.1 模块总览

| 项目 | 内容 |
| --- | --- |
| 模块目标 | 让玩家不用读配置，也能通过 UI 看懂技能、状态、伤害、治疗和战斗阶段变化。 |
| 画面关注点 | 技能栏、冷却/吟唱/持续施法状态、血条、状态图标、敌我飘字、阶段提示。 |
| 表现链路 | 战斗事件发生 → UI 状态响应 → 优先级判断 → 显示/合并/降级 → 状态收束。 |
| 前端关注 | UI 状态机、显示优先级、同屏信息合并、冷却/吟唱/禁用状态同步。 |
| 美术关注 | 图标状态、飘字字体与颜色、状态图标层级、阶段提示样式。 |
| 配套图示 | 技能 CD/吟唱 UI、状态进入/持续/结束、战场信息分区、特效遮挡与画面预算等后续图。 |
| 产出目标 | 前端能建立 UI 事件响应规则；美术能按优先级设计图标、飘字和提示样式。 |

#### 3.1.2 UI 信息响应流程

```mermaid
flowchart TD
    A["战斗事件发生"] --> B{"事件类型"}

    B --> B1["技能释放"]
    B --> B2["技能冷却"]
    B --> B3["伤害/治疗"]
    B --> B4["状态变化"]
    B --> B5["阶段变化"]

    B1 --> C1["技能图标：释放/吟唱/持续"]
    B2 --> C2["技能图标：冷却/即将就绪"]
    B3 --> C3["飘字 + 血条变化"]
    B4 --> C4["状态图标 + 战场状态反馈"]
    B5 --> C5["阶段提示/卡牌/奖励"]

    C1 --> D["按优先级显示"]
    C2 --> D
    C3 --> D
    C4 --> D
    C5 --> D

    D --> E{"是否与战场主信息冲突"}
    E --> E1["不冲突：正常显示"]
    E --> E2["冲突：降级/延迟/合并"]

    E1 --> F["UI 状态收束"]
    E2 --> F
```

#### 3.1.3 UI 分类

技能 UI：

- 技能冷却：图标灰显并显示倒计时。
- 即将释放：图标轻微高亮或呼吸效果。
- 正在释放：图标常亮并可能伴有施法动画。
- 吟唱/读条：图标上叠加读条进度。
- 持续施法：图标变化以表示维持状态。
- 释放成功：短暂的成功反馈，例如闪光。
- 被打断/失效：明确的打断或失效提示。
- 禁用/不可用：图标完全灰显且不可交互。

状态 UI：

- 己方 Buff：增益效果，显示在己方状态区。
- 己方 Debuff：减益效果，显示优先级高。
- 敌方 Buff：敌人增益，显示在目标状态区。
- 敌方 Debuff：敌人减益，显示优先级高。
- 控制状态：如眩晕、冻结，需特殊图标。
- 层数：可叠加状态的当前层数显示。
- 持续时间：状态剩余时间的倒计时。
- 状态刷新/结束：状态刷新或消失的提示。

战斗阶段 UI：

- 包含开战提示、波次提示、Boss 阶段提示、卡牌选择、奖励提示、战斗结束等。
- Boss 阶段提示规格最高。
- 卡牌选择是明确的节奏断点。
- 奖励/胜利表现需要清理战斗噪声。

### 3.2 屏幕信息分区

屏幕信息分区用于解决“信息放在哪里”。不同信息按照重要度、作用对象和打断程度放入不同区域，避免敌我飘字、状态、阶段提示和低优先级信息混在一起。

| 分区 | 主要内容 | 显示规则 |
| --- | --- | --- |
| 场上区（上半屏） | 敌方受伤、暴击、击杀、敌方 Buff/Debuff 变化 | 跟随敌方单位移动，玩家抬头即可看到“怪怎么了”。 |
| 角色区（下半屏） | 己方受伤、治疗、护盾、己方状态变化 | 跟随己方单位或固定在屏幕下方，玩家低头即可看到“自己人怎么了”。 |
| 中央提示区 | 开战、波次、Boss 转阶段、卡牌选择、战斗胜利/失败 | 用于打断注意力，拥有更高遮挡优先级，但必须短暂且可收束。 |
| 边缘区 | 危险来源提示、受击方向、屏幕边缘泛红/泛蓝 | 不遮挡中央视野，用于余光可见的危险或方向提示。 |
| 角落区（信息密度区） | DOT 小伤害、持续掉血、低优先级 P6 信息 | 信息过多时降级到角落或弱化显示，小字不抢注意力。 |

核心原则：越关键、越需要立刻理解的信息，越靠近屏幕中央；越次要、越偏持续确认的信息，越可以靠边角或弱化。中央提示区只用于战斗阶段、选择、结算等明确节奏断点，不能被普通伤害或普通状态频繁占用。

### 3.3 飘字与状态反馈

飘字用于结果确认，不承担全部战斗反馈。飘字位置遵循屏幕信息分区，不单独再建立一套分区系统。

#### 3.3.1 飘字落区规则

- 敌方结果飘字：使用场上区，位于敌人默认中心点上方，跟随敌方单位移动。
- 己方结果飘字：使用角色区，位于屏幕下半区或己方状态附近。
- 状态/特殊提示：优先靠近作用对象，必要时同步到状态 UI。
- 低优先级信息：DOT 小伤害、持续掉血、小额多段伤害等 P6 信息可降级到角落区。
- 中央提示区不承载普通飘字，只用于 Boss 转阶段、波次、卡牌选择、战斗结算等阶段信息。

#### 3.3.2 飘字颜色规则

- 敌方普通伤害：白色 / 浅黄色，默认，不抢夺高价值表现。
- 己方受伤：红色 / 橙红色，明确危险信号。
- 治疗/恢复：绿色，与伤害形成鲜明对比。
- 暴击/强命中：黄色 / 橙色 / 金色，高价值输出标识。
- 击杀：金色 / 白金色，可短暂使用更强效果。
- 格挡/减免：蓝灰色 / 白蓝色，表达防御成功。
- 闪避/未命中：灰色 / 白灰色，表达攻击落空。
- 免疫/抵抗：紫灰色 / 深灰色，表达完全无效。
- 状态变化：使用对应状态色，例如冰蓝色、毒绿色。

#### 3.3.3 显示优先级与动效强度

```text
P0 玩家受伤 / 被控：保留完整表现，明显弹出/抖动。
P1 击杀 / 暴击 / 破防：保留完整表现，放大 + 短促冲击。
P2 控制状态变化：保留完整表现，短文字 + 状态色。
P3 治疗 / 护盾 / 恢复：可简化，但要可读，小幅上浮。
P4 格挡 / 免疫 / 抵抗 / 反弹：可简化，稳定弹出。
P5 普通伤害：正常显示，小幅上浮 + 快速淡出。
P6 多段小伤害 / DOT 单跳：可合并、节流、缩小或延迟显示。
```

处理规则：

- P0-P2 必须保留完整表现。
- P3-P4 可适当简化但需保持可读性。
- P5 正常显示。
- P6 可合并、弱化或降级到角落区。
- 同一目标短时间内多次小伤害优先合并。

## 4. 结果表现模块

结果表现模块负责定义普通、强化、失败、防御、DOT、受击绑定等结果差异，避免所有命中看起来一样。

### 4.1 命中与结果路径

#### 4.1.1 模块总览

| 项目 | 内容 |
| --- | --- |
| 模块目标 | 定义普通结果、强结果、失败结果、群体结果之间的表现差异。 |
| 画面关注点 | 命中点、受击特效、飘字强度、音效/震动、死亡/清理反馈、Boss/特殊反馈。 |
| 表现链路 | 行动生效 → 结果判定 → 强弱/成功失败分级 → 对应表现路径 → UI 确认。 |
| 前端关注 | 命中结果枚举、表现优先级、失败结果路径、群体/多段/DOT 节流。 |
| 美术关注 | 普通/暴击/击杀/格挡/抵抗的特效差异，受击点、DOT 持续状态和强弱层级。 |
| 产出目标 | 前端能按结果类型触发不同表现；美术能知道哪些结果必须强化，哪些必须克制。 |

#### 4.1.2 命中与结果反馈流程

```mermaid
flowchart TD
    A["行动生效"] --> B["判定目标"]
    B --> C{"判定结果"}

    C --> C1["命中"]
    C --> C2["暴击/重命中"]
    C --> C3["格挡"]
    C --> C4["闪避/未命中"]
    C --> C5["免疫/抵抗"]
    C --> C6["击杀"]

    C1 --> D1["普通命中点 + 小反馈"]
    C2 --> D2["强化命中点 + 强反馈"]
    C3 --> D3["格挡反馈"]
    C4 --> D4["失败反馈"]
    C5 --> D5["无效反馈"]
    C6 --> D6["死亡/奖励反馈"]

    D1 --> E["血量变化"]
    D2 --> E
    D3 --> E
    D4 --> F["无正常命中后果"]
    D5 --> F
    D6 --> G["清理目标/进入奖励或下一目标"]

    E --> H["飘字/UI确认"]
    F --> H
    G --> H

    H --> I["战斗继续"]
```

#### 4.1.3 结果路径规则

- 命中、重命中/暴击、格挡、未命中/闪避、免疫/抵抗、击杀必须使用不同表现路径。
- 强化结果需要在命中点特效、受击反馈、音效、飘字等多个维度增强。
- 失败结果禁止播放成功命中表现，必须使用独立的失败或无效表现。
- 击杀表现需要包含目标清理，并自然过渡到奖励结算或切换下一目标。

### 4.2 强弱分级

| 结果类型 | 表现强度要求 | 动效增强 | 音效增强 | 飘字增强 |
| --- | --- | --- | --- | --- |
| 普通命中 | 克制，为高价值事件留空间 | 小幅命中点特效 | 基础命中音效 | 普通颜色，标准大小 |
| 重命中/暴击 | 显著强化，突出价值 | 强化命中点 + 可能全屏闪白 | 厚重、清脆的特殊音效 | 放大、金色/橙色，特殊字体 |
| 击杀 | 短暂但强烈的终结反馈 | 爆炸、消散等死亡特效 + 可能慢放 | 终结技、胜利音效 | 最大、白金/金色，“击杀！” |
| Boss 有效命中 | 高于普通命中，低于阶段事件 | 更明确的命中点或破防表现 | 更厚重的反馈 | 可强化但不抢阶段提示 |

强弱分级要求：

- 强弱差异不能只靠数字大小。
- 重命中以上结果需要在动作、特效、音效、飘字中至少选择两项增强。
- 普通命中必须克制，为高价值事件留空间。

### 4.3 成功防御

成功防御包括格挡、闪避、护盾吸收、招架等结果。它们虽然不造成正常伤害，但对防御方来说是“成功防住了”，需要正向表达。

| 结果类型 | 表现定位 | 动效增强 | 音效增强 | 飘字增强 |
| --- | --- | --- | --- | --- |
| 格挡 | 防御成功 | 盾牌、格挡光效，无受击抖动 | 金属撞击、防御成功音效 | 蓝色/灰色，“格挡” |
| 闪避/未命中 | 攻击落空 | 轻微偏移特效，无命中点 | 落空音效 | 灰色，“闪避”“未命中” |
| 免疫/抵抗 | 完全无效 | 护盾、光环等无效化特效 | 低沉、无效化音效 | 紫色/深灰，“免疫”“抵抗” |
| 护盾吸收 | 伤害被吸收 | 护盾波纹、短促亮边 | 护盾吸收音效 | 蓝白色，“吸收” |

规则：

- 成功防御不能播放完整成功命中表现。
- 格挡、闪避、护盾吸收等结果需要给防御方正反馈。
- 抵抗、免疫等无效结果需要与正常命中明确区分。
- 反弹需要同时表达“来源”和“返回结果”。

### 4.4 DOT/持续伤害

DOT 不同于普通多段攻击。多段攻击强调一次技能内的连续命中节奏，DOT 强调一个状态在一段时间内持续生效。玩家需要感知“目标正在持续掉血/受影响”，而不是看到一堆小数字。

| 项目 | 规则 |
| --- | --- |
| 状态视觉 | 必须有持续状态表现，例如毒雾、灼烧、冰霜、蛛网、腐蚀边缘或 Debuff 图标呼吸。 |
| 飘字频率 | 默认低频显示，可按固定间隔抽样显示，或在多跳后合并为一次数字。 |
| 状态来源 | 颜色应跟状态类型绑定，例如毒为绿色、燃烧为橙红、冰冻为冰蓝。 |
| 开始/结束 | DOT 进入时需要状态施加反馈；结束、刷新或叠层变化时需要 UI 或轻量战场反馈确认。 |
| 优先级 | DOT 单跳默认按 P6 处理；当 DOT 导致击杀、破盾、触发连锁效果时，升级到对应高价值结果路径。 |
| 镜头/节奏 | DOT 单跳不触发镜头或节奏断点；关键结果升级后按全局表现调度处理。 |

### 4.5 受击绑定点

为简化实现并保证一致性，当前阶段采用统一的受击点绑定规则。

| 对象/情况 | 绑定规则 |
| --- | --- |
| 普通小怪、中型怪、精英怪 | 默认使用模型中心点播放所有受击表现，包括特效、飘字、音效。 |
| 中大型怪 | 若中心点位置不合适，优先通过 X/Y/Z 轴偏移修正，而非新增复杂挂点。 |
| 群体技能 | 可拥有独立范围主特效，但每个实际命中的单体目标，受击反馈仍基于各自默认中心点。 |
| Boss/特殊怪 | 仅预留特殊受击点概念，不作为当前普通怪实现要求。 |
| 失败/减免结果 | 格挡、免疫、抵抗等可复用默认中心点，但表现强度与视觉样式必须区别于正常命中。 |

## 5. 全局表现调度

全局表现调度负责定义玩家如何“观看”战斗，以及哪些事件应该打破自动战斗的均匀节奏。它不替代动作、动效、结果和 UI 反馈，而是在高价值事件发生时提升观看权重、节奏层次和战斗段落感。

### 5.1 模块总览

| 项目 | 内容 |
| --- | --- |
| 模块目标 | 避免自动战斗从头到尾保持同一观看节奏，让关键事件被玩家明确感知。 |
| 画面关注点 | 战场构图、目标位置、命中点、Boss 阶段变化、击杀清理、UI 与飘字可读性。 |
| 表现链路 | 战斗事件触发 → 事件价值判断 → 镜头/节奏调度 → 关键反馈展示 → 回归常规战斗。 |
| 前端关注 | 事件优先级、镜头触发条件、顿帧/暂停时长、同屏事件合并、镜头状态回正。 |
| 美术关注 | 镜头强调幅度、命中冲击感、阶段断点演出、击杀收束、强弱事件差异。 |
| 配套图示 | 后续可补充镜头调度优先级图、战斗节奏峰谷图、Boss 转阶段节奏图。 |
| 产出目标 | 前端能按事件价值触发镜头/节奏规则；美术能知道哪些事件需要更高规格的观看表现。 |

### 5.2 镜头调度

镜头调度层负责处理镜头推进、拉远、轻微跟随、震动、特写、回正等观看方式。

规则：

- 普通事件以稳定观看为主。
- 高价值事件可以触发轻量镜头强调。
- Boss 转阶段、关键击杀、玩家濒死等事件可以使用更高规格镜头调度。
- 镜头强化必须服务信息理解，不能遮挡技能 UI、飘字或关键受击反馈。
- 镜头必须短促可恢复，不能让战斗长时间停在演出态。

### 5.3 战斗节奏

战斗节奏层负责处理加速、减速、顿帧、短暂停顿、慢动作和节奏断点。

规则：

- 普通命中不触发明显顿帧。
- 重命中、暴击、强攻击命中可以使用短促节奏强化。
- 击杀和 Boss 转阶段可以形成明确节奏断点。
- 卡牌选择、奖励选择、战斗胜利等节点需要从战斗噪声中收束出来。
- 同一时间多个高价值事件触发时，需要合并节奏反馈，避免连续停顿。

### 5.4 事件触发分级

| 事件层级 | 事件类型 | 镜头规则 | 节奏规则 |
| --- | --- | --- | --- |
| 常规事件 | 普通命中、普通治疗、DOT 单跳 | 不触发或极弱反馈 | 保持原节奏 |
| 中价值事件 | 重命中、暴击、成功防御、强攻击命中 | 可轻微推进、轻震或命中点强调 | 可短促顿帧或节奏加重 |
| 高价值事件 | 击杀、关键控制、反弹关键结果、DOT 击杀 | 可短促镜头强调或目标清理跟随 | 可形成短暂停顿或清理节奏 |
| 阶段事件 | Boss 转阶段、卡牌选择、奖励选择、战斗胜利 | 可拉远、居中、回正或阶段镜头 | 必须形成明确节奏断点 |

### 5.5 使用约束

- 稳定优先：自动战斗的基础观看状态必须稳定，不能因为频繁镜头变化导致玩家失去战局理解。
- 高价值才强化：普通伤害、普通 DOT 单跳、普通小怪死亡不触发强镜头或强节奏。
- 信息不遮挡：镜头和节奏强化不能压住技能 UI、状态 UI、关键飘字和受击确认。
- 合并不叠加：同一时间多个事件触发时，按最高价值事件执行一次镜头/节奏反馈。
- Boss 可升级：Boss 战允许更高规格的镜头和节奏调度，但仍需要保留玩家对场上状态的连续理解。

### 5.6 与其他模块的关系

- 己方行动表现触发：关键技能释放、暴击、击杀、Buff 强化首次释放。
- 敌方行动表现触发：强攻击命中、控制命中、Boss 技能、玩家濒死。
- UI 信息表现触发：Boss 阶段提示、卡牌选择、奖励结算、战斗结束。
- 结果表现触发：重命中、成功防御、反弹、DOT 击杀、免疫关键技能等高价值结果。

最终规则：镜头和节奏只强化关键事件，不承担基础信息说明。基础信息仍由动作、动效、结果反馈和 UI 完成。
