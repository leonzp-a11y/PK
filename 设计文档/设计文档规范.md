# PK 设计文档规范

本文档用于统一项目内设计文档的编写结构。所有系统、玩法、世界观剧情、逸闻、活动、功能模块等设计文档，优先遵循本规范。

## 1. 设计目的

设计文档开篇必须先定义该设计的目的、提供给玩家的体验，以及目标设计。

### 1.1 概念简述

用简洁文字说明该设计是什么。

- 一句话说明该设计的核心概念。
- 避免过早展开细节，先让团队快速理解设计方向。

### 1.2 玩家侧体验

说明玩家在该设计中会获得什么体验，以及希望达成什么预期效果。

- 玩家会看到什么。
- 玩家会做什么。
- 玩家会感受到什么。
- 预期达到的效果是什么。

预期效果可以由主策、系统负责人或提出该设计的人给出。后续设计应以该预期效果为主要依据，判断功能取舍、体验优先级和设计是否偏题。

### 1.3 目标设计

说明该系统或玩法在不同阶段希望达成的目标。

- 短期目标：上线初期要解决什么问题，提供什么基础体验。
- 中期目标：版本迭代中要扩展什么内容，增强什么循环。
- 长期目标：该设计在项目长期运营中承担什么价值，是否具备持续扩展空间。

## 2. 设计思维导图

设计思维导图用于定义该设计的具体内容，并按系统模块拆分。

要求：

- 按模块拆解，不按零散想法堆叠。
- 文字简洁，逻辑清晰。
- 能看出模块之间的从属关系、关联关系和设计边界。
- 若设计对象是世界观、剧情、逸闻等内容，也应拆分为搭建该系统所需的不同模块。

建议结构：

```text
设计名称
├─ 核心体验
├─ 系统入口
├─ 玩法流程
├─ 内容结构
├─ 奖励与消耗
├─ 成长或收集
├─ 展示与反馈
├─ 关联功能
└─ 配置需求
```

具体项目可按实际设计调整模块，但应保持层级清楚。

## 3. 设计交互原型图

设计交互原型图用于表达该设计的基础 UI 设计、信息布局与交互关系。

需要覆盖：

- 该设计的主要界面。
- 系统入口与跳转关系。
- 核心操作路径。
- 关键状态展示。
- 与其他系统关联的界面或入口。

原型图不要求达到最终美术品质，但必须能说明：

- 玩家从哪里进入。
- 玩家在界面中看到什么信息。
- 玩家可以进行哪些操作。
- 操作后会进入什么状态或界面。

## 4. 具体设计

当设计目的、思维导图、交互原型图经过策划内部会议确认并修正后，再进入详细功能文档阶段。

具体设计应按模块拆分撰写，常见模块包括：

- 前端表现。
- 后端规则。
- 交互逻辑。
- 功能逻辑。
- 数据流转。
- 状态变化。
- 异常情况。
- 埋点或日志需求。

具体设计要求：

- 明确规则，不只描述感觉。
- 明确边界条件。
- 明确不同状态下的表现。
- 明确功能之间的依赖关系。
- 明确需要程序、美术、交互、数值、剧情等岗位配合的内容。

## 5. 配置结构

根据设计文档中的功能需求，设计对应的配置结构。

配置结构需要包含：

- 配置表名称。
- 配置用途。
- 主键定义。
- 字段名称。
- 字段类型。
- 字段说明。
- 填写规则。
- 示例值。
- 关联表或关联字段。

建议字段说明格式：

| 字段名 | 字段类型 | 是否必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| id | int | 是 | 唯一 ID | 1001 |
| name | string | 是 | 名称 | 示例名称 |
| unlock_condition | string | 否 | 解锁条件 | player_level >= 10 |

配置结构应服务于具体设计，避免提前设计无用途字段。

## 标准文档模板

新建设计文档时，可复制以下结构：

```markdown
# 设计名称

## 1. 设计目的

### 1.1 概念简述

### 1.2 玩家侧体验

### 1.3 目标设计

#### 短期目标

#### 中期目标

#### 长期目标

## 2. 设计思维导图

## 3. 设计交互原型图

## 4. 具体设计

### 4.1 前端表现

### 4.2 后端规则

### 4.3 交互逻辑

### 4.4 功能逻辑

### 4.5 异常情况

## 5. 配置结构

### 5.1 配置表列表

### 5.2 字段定义

### 5.3 配置示例
```
