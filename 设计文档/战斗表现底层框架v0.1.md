# 战斗表现底层框架 v0.1

## 1. 文档目标

本文档用于建立一套跨武器、跨法术、跨怪物、跨战斗事件都可复用的战斗表现底层规则。它不是某一个技能的表现清单，而是所有战斗表现进入制作前必须遵循的基础协议。

该框架服务于当前战斗 demo 的持续迭代，尤其针对录屏中暴露出的核心问题：

- 特效强度足够，但命中对象、接触点和战斗结果容易被遮挡。
- 敌人和 Boss 受击后果感不足，命中主要依赖数字表达。
- 敌方控制和伤害技能的前摇、来源、危险边界不够清晰。
- 第一人称武器/法杖有身份感，但容易长期占据命中阅读区。
- 多敌人战斗中，VFX、飘字、UI 和敌人血条同时叠加，画面层级被拉平。

本文档优先定义底层规则，再允许在其上扩展具体武器、法术、怪物、Boss、状态、卡牌升级和局内成长表现。

## 2. 核心原则

所有战斗表现必须优先回答以下 6 个问题：

1. 谁发起了这次事件？
2. 这次事件的意图是什么？
3. 它影响了谁？
4. 它什么时候生效？
5. 它造成了什么结果？
6. 玩家下一步应该如何理解和行动？

如果一个战斗表现不能回答这 6 个问题，它就会变成“画面上发生了事情”，但玩家无法稳定判断战斗状态。

## 3. 三层结构

建议将现有“技能系统”和“技能表现”拆成三层：

```text
技能系统层
├─ 技能类型
├─ 释放方式
├─ 目标/范围
├─ 实际效果
├─ 资源/CD/条件
└─ 技能学习/成长

战斗事件层
├─ 事件发起者
├─ 事件目标
├─ 事件类型
├─ 生效阶段
├─ 命中类型
├─ 威胁等级
├─ 表现优先级
└─ 反馈通道组合

表现资产层
├─ 动作
├─ 特效
├─ 音效
├─ 飘字
├─ UI
├─ 镜头
└─ 状态残留
```

其中“战斗事件层”是核心。任何武器、法术、怪物技能、Boss 技能、被动触发、环境伤害，都应先转化为战斗事件，再根据事件规则调用表现资产。

## 4. 战斗事件标准结构

每个战斗事件至少包含以下字段：

| 字段 | 说明 | 示例 |
|---|---|---|
| 发起者 | 谁触发事件 | 玩家、普通敌人、精英敌人、Boss、召唤物、环境 |
| 目标 | 事件影响谁 | 单体敌人、全体敌人、玩家自身、前排、后排、区域 |
| 事件类型 | 事件意图 | 攻击、治疗、增益、减益、控制、召唤、防御、反击 |
| 生效方式 | 什么时候生效 | 即时、延迟、飞行命中、持续区域、周期触发、条件触发 |
| 命中类型 | 结果如何表现 | 轻命中、重命中、暴击、格挡、闪避、控制命中、击杀 |
| 威胁等级 | 对玩家理解的优先级 | 普通、强提示、危险、致命、控制 |
| 反馈通道 | 调用哪些表现资产 | 动作、VFX、SFX、UI、飘字、镜头、状态残留 |

底层规则：

- 所有战斗事件必须明确“发起者、目标、结果”。
- 所有会伤害或控制玩家的敌方事件，必须明确“预兆、生效、结果”。
- 所有玩家主动输出事件，必须明确“响应、命中、后果”。

## 5. 战斗表现阶段协议

所有战斗事件按以下感知链路组织：

```text
意图预告
-> 动作承诺
-> 效果生成
-> 飞行/扩散/持续
-> 接触/命中
-> 结果确认
-> 后果表现
-> 恢复/进入下一状态
```

对应表现层：

```text
敌我动作
-> 施法/攻击动作
-> 起手特效
-> 飞行/区域特效
-> 命中特效
-> 受击动作
-> 飘字/UI/音效
-> 状态残留
-> 冷却/节奏回收
```

不是所有事件都必须完整占满每个阶段，但任何被玩家感知为“重要”的事件，都不应跳过“预告、接触、结果、后果”这四类核心信息。

## 6. 信息优先级

战斗表现的画面、声音、UI 必须遵循以下优先级：

| 优先级 | 信息 | 表现目标 |
|---|---|---|
| P0 | 生存威胁 | 敌人即将攻击、控制、秒杀、范围危险 |
| P1 | 玩家行为确认 | 玩家输入后，系统是否响应 |
| P2 | 命中对象 | 技能打中了谁，没打中谁 |
| P3 | 战斗结果 | 伤害、治疗、格挡、闪避、暴击、死亡、状态变化 |
| P4 | 后果感 | 敌人被打断、后退、冻结、破防、倒地 |
| P5 | 氛围强化 | 光效、粒子、屏幕震动、环境装饰 |

底层规则：

- P5 不得遮挡 P0-P4。
- 大面积特效不能抢走目标、命中、危险、结果四类核心信息。
- 同一时刻屏幕中央只能有一个主信息。

## 7. 命中类型规则

不同武器和法术可以风格不同，但必须挂接到统一的命中类型规则。

### 7.1 轻命中

适用：普通攻击、小伤害、多段快速命中。

必须包含：

- 小型接触点。
- 短促受击闪白或描边。
- 小数字或轻量结果提示。

避免：

- 大面积屏幕特效。
- 过强镜头震动。
- 长时间遮挡敌人轮廓。

### 7.2 重命中

适用：重击、蓄力攻击、强法术、关键打击。

必须包含：

- 明确接触爆点。
- 短暂停顿或节奏强调。
- 明显受击动作、硬直或局部反应。
- 更高权重的音效与数字。

### 7.3 暴击

适用：暴击、弱点命中、特殊增伤。

必须包含：

- 与普通命中不同的颜色或符号。
- 数字放大或弹出节奏差异。
- 特殊音效瞬态。

注意：

- 暴击表现必须强于普通命中，但不能强于击杀、破防、Boss 阶段变化。

### 7.4 格挡

适用：敌人格挡、玩家格挡、护盾吸收。

必须包含：

- 防御火花或护盾波纹。
- “格挡/吸收/减免”类结果提示。
- 与正常命中不同的受击反应。

底层规则：

- 格挡不能表现得像正常命中。
- 如果没有造成完整伤害，就不能播放完整受击后果。

### 7.5 闪避/未命中

适用：闪避、失手、目标移出范围、免疫。

必须包含：

- “未命中/闪避/免疫”结果文字或图标。
- 无正常命中爆点。
- 目标可有轻微偏移动作。

### 7.6 控制命中

适用：冻结、蛛网、眩晕、击飞、束缚、沉默。

必须包含：

- 状态进入表现。
- 状态持续表现。
- 状态结束表现。
- UI 状态图标或倒计时。

底层规则：

- 状态类效果不能只有进入表现。
- 玩家必须知道状态是否仍在持续，以及何时结束。

### 7.7 击杀

适用：敌人死亡、Boss 阶段结束、召唤物消失。

必须包含：

- 死亡动作或消散动作。
- 奖励/掉落/阶段确认。
- 画面清理，避免死亡残留遮挡下一轮战斗。

## 8. 敌方威胁表现规则

所有敌方伤害或控制事件必须遵循：

```text
预兆 -> 生效 -> 结果
```

### 8.1 预兆

用于告诉玩家“危险要来了”。

可用通道：

- 敌人姿态变化。
- 身体部位发光。
- 地面范围提示。
- 屏幕边缘提示。
- 声音预警。
- UI 警告。

### 8.2 生效

用于告诉玩家“危险正在发生”。

可用通道：

- 弹道飞行。
- 范围扩散。
- 命中点。
- 全屏或局部覆盖。
- 玩家动作受限。

### 8.3 结果

用于告诉玩家“你受到了什么影响”。

可用通道：

- 玩家掉血。
- 红屏或方向提示。
- 状态图标。
- 控制倒计时。
- 技能按钮禁用/变灰。

底层规则：

- 所有敌方控制必须至少提前 0.4-0.7 秒给出可读预兆。
- 预兆不能只靠敌人血条上方的小图标承担。
- 玩家受伤必须能看出来源方向或状态来源。

## 9. 反馈通道组合规则

反馈通道包括：

```text
动作
特效
音效
UI
飘字
镜头
状态残留
环境变化
```

### 9.1 通道数量规则

| 事件等级 | 推荐突出通道数 | 示例 |
|---|---:|---|
| 普通事件 | 2-3 个 | 普通攻击、小治疗、小状态刷新 |
| 重要事件 | 3-4 个 | 重击、冻结、精英怪攻击 |
| 高价值事件 | 4-5 个 | 暴击、破防、击杀、Boss 技能 |
| 阶段事件 | 5 个以上 | Boss 入场、阶段转换、大招、胜利 |

底层规则：

- 普通事件不要全通道强化。
- 高价值事件必须和普通事件拉开表现差异。
- 同屏多个事件同时发生时，只有一个事件能成为主表现。

### 9.2 通道分工

| 通道 | 主要职责 |
|---|---|
| 动作 | 表示意图、承诺、受击后果 |
| 特效 | 表示能量、范围、接触、状态 |
| 音效 | 表示瞬态、材质、危险、奖励 |
| UI | 表示规则、状态、数值、限制 |
| 飘字 | 表示结果，但不能独立承担全部反馈 |
| 镜头 | 表示重量、冲击、阶段强调 |
| 状态残留 | 表示持续影响和区域占用 |

## 10. 画面预算规则

画面应按区域进行预算管理。

### 10.1 中央命中区

优先级：

1. 敌人剪影。
2. 命中点。
3. 血条和状态。
4. 伤害/结果文字。
5. 主特效。
6. 氛围残留。

规则：

- 中央命中区同时只能有一个主特效。
- 持续 VFX 必须让出敌人剪影。
- 命中特效可以亮，但持续时间要短。

### 10.2 下方操作区

承载：

- 技能冷却。
- 玩家血量/能量。
- 状态图标。
- 可用道具。

规则：

- 全屏 VFX 不得长期遮挡技能冷却。
- 玩家被控时，按钮状态变化必须比装饰特效更清楚。

### 10.3 第一人称武器区

承载：

- 武器身份。
- 施法动作。
- 输入响应。

规则：

- 武器/法杖可以在释放前进入中心。
- 释放后必须回到安全区。
- 命中瞬间不得长期遮挡敌人血条和接触点。

### 10.4 全屏覆盖

适用：

- 玩家被控。
- Boss 大招。
- 阶段转换。
- 胜利/失败。

规则：

- 全屏覆盖必须有明确原因。
- 全屏覆盖必须有明确持续时间。
- 全屏覆盖不得遮挡解除方式或核心状态。

## 11. 状态表现规则

所有状态必须包含三段：

```text
进入 -> 持续 -> 结束
```

### 11.1 进入

告诉玩家状态刚刚发生。

示例：

- 冻结：冰晶从命中点扩散。
- 中毒：绿色液体或毒雾附着。
- 蛛网：网线从边缘收束到目标。
- 增益：角色周围出现短促上升光效。

### 11.2 持续

告诉玩家状态仍然存在。

示例：

- 状态图标。
- 低强度循环特效。
- 敌人材质变化。
- 倒计时或层数显示。

### 11.3 结束

告诉玩家状态已经解除。

示例：

- 冰壳破碎。
- 毒液蒸发。
- 蛛网断裂。
- 状态图标消失。

底层规则：

- 状态持续表现必须低于命中表现。
- 状态结束必须清楚，否则玩家会误判当前战斗规则。

## 12. Boss 表现规则

Boss 可以不被击退，但必须被影响。

### 12.1 Boss 入场

必须建立：

- 体型压迫。
- 危险身份。
- 主要攻击主题。
- 战斗阶段感。

### 12.2 Boss 受击

Boss 不适合像小怪一样频繁位移，但必须有局部反馈：

- 头部后仰。
- 腹部震动。
- 腿部收缩。
- 甲壳裂光。
- 眼睛闪烁。
- 局部冻结/燃烧/中毒。

### 12.3 Boss 大招

必须经历：

```text
强预兆 -> 明确危险边界 -> 生效冲击 -> 结果确认 -> 恢复窗口
```

### 12.4 Boss 阶段变化

可调用：

- 镜头节拍。
- 背景变化。
- UI 阶段提示。
- 新弱点或新状态。
- 技能表现升级。

## 13. 元素/武器扩展规则

底层框架不直接绑定某一种风格，但每种元素和武器应在底层规则上定义自己的表现语言。

### 13.1 冰系

核心语言：

- 冷、硬、冻结、破碎、控制、减速。

应优先表现：

- 命中点冰晶。
- 目标局部冻结。
- 冰壳持续。
- 破碎结束。

避免：

- 长时间白雾遮挡敌人。
- 所有冰系技能都使用同等强度蓝白爆光。

### 13.2 火系

核心语言：

- 热、爆发、灼烧、持续伤害、扩散。

应优先表现：

- 接触火花。
- 燃烧附着。
- 火焰跳动。
- 灼烧结束。

### 13.3 毒系

核心语言：

- 腐蚀、持续、危险区域、削弱。

应优先表现：

- 来源明确。
- 持续区域边界。
- 中毒图标和层数。
- 颜色与治疗/增益区分。

### 13.4 物理武器

核心语言：

- 方向、材质、重量、接触。

应优先表现：

- 攻击轨迹。
- 接触火花。
- 受击方向。
- 击退/硬直。

## 14. 与当前技能表现框架的衔接

现有框架中的“动作、特效、飘字、音效、技能冷却表现”应继续保留，但它们应作为表现资产层，而不是表现设计的起点。

推荐改造方式：

```text
技能定义
-> 生成战斗事件
-> 判断事件等级与命中类型
-> 选择反馈通道组合
-> 调用动作/特效/音效/飘字/UI/镜头
-> 进行画面预算与优先级检查
```

这样可以避免每个技能各自堆表现，也能保证不同系统扩展后仍然遵循同一套可读性和反馈规则。

## 15. v0.1 硬规则清单

第一版建议先执行以下硬规则：

1. 所有战斗事件必须有“发起者、目标、结果”。
2. 所有敌方伤害/控制必须有“预兆、生效、结果”。
3. 所有玩家命中必须有“接触点、目标反馈、结果确认”。
4. 所有状态类效果必须有“进入、持续、结束”三段表现。
5. 所有大面积特效不得遮挡敌人剪影、血条和玩家核心 UI。
6. 所有 Boss 受击必须表现“被影响”，即使不击退也要有局部反应。
7. 普通事件最多突出 2-3 个反馈通道，高价值事件才允许多通道强化。
8. 第一人称武器/法杖可以强化身份感，但释放后必须让出命中阅读区。
9. 同一时刻屏幕中央只能有一个主表现事件。
10. 飘字只能作为结果确认，不能代替动作、特效和受击后果。

## 16. 后续扩展方向

在底层框架稳定后，建议继续拆分以下专题：

- 冰系表现规范。
- 火系表现规范。
- 毒系表现规范。
- 物理武器表现规范。
- 蜘蛛怪威胁表现规范。
- Boss 表现规范。
- 卡牌升级后的技能表现强化规则。
- 战斗 UI 与飘字规范。
- 音效分层与混音优先级规范。
- 录屏验收检查表。
